# Project Documentation

## このファイルの役割

- このファイルには、プロジェクト横断の運用ルールと参照先だけを書く
- 仕様、UI 詳細、データ定義、TODO をここに重ね書きしない
- 実装済みの挙動はソースコードを正とし、`docs/` は運用上必要な参照先と合意事項だけを書く
- ドキュメントの一覧と責務は [docs/index.md](docs/index.md) を一次ソースにする

## 必須ルール

- 仕様変更によって、文書化されたプロダクト意図、不変条件、責務分担、運用手順が変わる場合だけ、同じ作業で対応するドキュメントを更新する。実装詳細だけの変更ではドキュメントを更新しない
- 実装とドキュメントがズレた状態を見つけたら、実装を正とする。ズレた記述が実装詳細の写経なら修正ではなく削除し、文書の責務に属する内容だけ残す
- 更新先ドキュメントは [docs/index.md](docs/index.md) から確認する
- ルート一覧、API入出力例、DB列、設定値一覧、環境変数一覧、UI項目一覧を Docs に複製しない。必要な読者を対応するコード上の一次ソースへ案内する
- このファイルを更新するのは、運用ルール、前提、参照先の意味が変わったときだけにする
- セキュリティ制約は [docs/security-invariants.md](docs/security-invariants.md) を常時優先する
- 確定できない値を「それっぽい既定値」で埋めて先に進めない。設定・外部API応答・DB値・権限・所有者・状態が確定できない場合は、明示的なエラー、未設定状態、確認導線のいずれかで扱う
- 明示的な指示がない限り、Git の commit / push は行わない
- `.codex/skills/**/SKILL.md` には、汎用的な呼び出し条件、標準入口、実行上の最小手順だけを書く。プロダクト仕様、UI 仕様、API 契約、DB 定義、暫定補正を重ね書きしない
- 新機能追加時は、Skill に局所分岐を足す前に、既存の標準フロー、`docs/`、実装側へ吸収する。Skill を変更するのは、汎用的な運用フローや入口の意味が変わった場合だけにする
- 通常サンドボックスでは Docker socket（例: `/Users/pc111/.docker/run/docker.sock`）へ接続できず、`docker` や `./vendor/bin/sail` が `operation not permitted` や `Docker or Podman is not running.` を返すことがある。この表示を見ても Docker 停止と即断しない。Docker / Sail の確認・テスト実行が必要な場合は、権限付きで `docker info` または `./vendor/bin/sail ps` を再実行して切り分け、必要な確認コマンドも権限付き Sail で再実行する

## 設計判断の原則

- 新機能や仕様判断の前に [docs/product-principles.md](docs/product-principles.md) と、導線へ影響する場合は [docs/user-flows.md](docs/user-flows.md) を確認する
- 要望を実装手段へ直訳せず、利用者の目的と既存導線のどこに置くかを先に決める
- 「データをどこまで保存するか」「最初に何を見せるか」「利用者がどこまで変更できるか」「非同期処理の対象は何か」を別概念として扱う。これらを実装都合で混同しない
- テストは実装の偶然ではなく、利用者体験上の不変条件に置く。初期状態、操作後の状態遷移、表示内容の一貫性、権限・所有権・失敗時の挙動を優先して確認する
- 解釈が複数あり、選択によって利用者の導線、公開範囲、保存対象、権限、非同期処理の範囲が変わる場合は、実装前に確認する
